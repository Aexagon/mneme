Mneme Greek: memory v0.3.0

The memory Claude Code keeps between sessions.

Every new chat starts from zero. Mneme loads what past chats learned into each session, then writes new, durable learnings back, so the next chat starts further along instead of starting over.

/mneme:remember save a learning /mneme:recall search it /mneme:review approve the inbox /mneme:loop drive a goal

the one idea to hold onto

How the memory works

The whole thing is a loop. A chat happens, useful things get noticed, they wait for your say-so, and once you approve them the next chat already knows. Here is the whole cycle, start to finish.

1

You have a chat

You work with Claude Code as usual. Nothing about your workflow changes.

2

Notes get taken

When the chat ends, Mneme quietly reads it and drafts short notes of anything durable and reusable — a preference, a fix, a fact.

3

They wait in an inbox

Drafts land in a waiting room. They are not trusted yet, and they are never shown to a future chat.

4

You approve them

You run /mneme:review and decide, one by one: keep it or throw it away. This is the gate.

5

They enter the notebook

Approved notes move into the trusted notebook. Discarded ones are gone.

6

The next chat knows

Every chat after that opens already knowing the approved notes. The loop begins again.

Nothing enters your trusted memory without your approval. Auto-drafted notes only ever sit in the waiting room until you say yes. That is the whole promise: the machine proposes, you decide.

the shape of it

Two tiers, one engine.

Mneme keeps two kinds of memory on the same simple machinery, and knowing which is which is the whole map. They pull in opposite directions: one stays small so it can load into every chat, the other is free to grow because it never loads uninvited.

the whole toolkit

Seven commands, and when to reach for each

You will use two of these most days and the rest now and then. If you only remember one, remember /mneme:review — that is the habit that makes the whole thing work.

/mneme:remember Save a learning right now. You tell it something worth keeping and it files a note on the spot — no waiting room, no approval needed, because you said it on purpose.
/mneme:recall Search your saved notes for something you half-remember telling it. Add --wiki <name> to search a knowledge wiki instead of your everyday memory.
/mneme:status A quick health check. How much is saved, whether anything is waiting for you to review, and whether the memory is getting too big for its own good.
the weekly habit /mneme:review Approve or discard the notes waiting in the inbox. This is the one habit to build — a couple of minutes, once a week. Nothing reaches trusted memory until you do it.
/mneme:lint A deeper tidy-up than status. It finds broken links between notes, notes nothing points to, and any two notes that contradict each other — and fixes them only after you confirm.
/mneme:ingest Add a whole document — a file or a web page — into a knowledge wiki. Mneme reads it, summarizes it into pages, and files it so you can ask questions about it later.
/mneme:loop Give it a goal and a "done" test. It keeps working, checking its own result, and retrying until the goal is met or it honestly reports it is stuck — saving what it learns along the way.

in practice

What people do with it

The pattern is the same each time: tell Claude once, and every future session already knows.

Developershouse rules that stick
  • Use pnpm, tests are vitest in __tests__, never touch /legacy. Said once, known in every future session on that repo. No more re-explaining the house rules after each fresh start.
  • The flaky CI test is a timezone bug. Pin TZ=UTC in setup. Six weeks later it resurfaces and the fix is already there, instead of being re-derived from scratch.
  • Terse answers, run typecheck before declaring done, deploy via make ship. A preference that follows you into every repo, not one project's config file.
  • Commit the project memory into the repo. Now every teammate's Claude inherits the same hard-won context. Onboarding that ships with the code.
Power usersnon-developers who live in Claude Code
  • Nicolette's voice is warm, no jargon. Banned words: synergy, leverage. Every session about that client starts knowing it. Brand rules and standing facts stop getting re-typed.
  • Our packages, pricing, and who's who. Captured once and carried into every chat about the business.
  • Ingest these 40 research papers into a wiki, then let me ask across them. Now the knowledge tier holds the whole stack. Ask a question, get an answer drawn from all of them, with citations — read only when you ask, never taxing your everyday chats.
Anyonea different axis: iteration, not memory
  • /mneme:loop "make the suite green" --done "npm test exits 0" Drives one goal to completion in the current chat: plan, act, verify, repeat against a criterion you state, learning each pass and stopping honestly when it is done or stuck.

from install to habit

Getting started

Three lines to install, then one small ritual to keep. Setting it up takes a minute; the payoff builds over the weeks that follow.

1

Add the marketplace and install

In Claude Code, run these two commands:

/plugin marketplace add Aexagon/mneme /plugin install mneme@mneme
2

Restart the session

Start a fresh chat. At the top you should see the Mneme memory load in — that line at the top is proof it is working. On day one it will be nearly empty, which is exactly right.

3

Make review a weekly ritual

Put /mneme:review on your Monday checklist. A couple of minutes to approve the notes that piled up over the week is all it takes to keep the memory sharp and trustworthy.

week 1

Feels normal. Little is saved yet, so chats behave the way they always have. That is expected — it is filling the waiting room.

week 2 and on

As you approve notes each week, chats start already knowing your context — your preferences, your projects, the fixes you found once. Every week it knows a little more.

plain files, your machine, nobody else's

Where your data lives

Everything Mneme knows is stored as plain text files on your own computer. You can open them, read them, edit them, or delete them yourself. Nothing is uploaded, synced to a cloud, or sent anywhere.

~/.claude/mneme/inbox/ the waiting room

Where auto-drafted notes sit before you approve them. Nothing here is trusted and nothing here is ever shown to a chat. /mneme:review empties it.

~/.claude/mneme/cache/ the trusted notebook

Your everyday memory — the notes you saved or approved. This is what loads into every chat. Kept small on purpose so it always fits.

~/.claude/mneme/wiki/ read on demand

The knowledge wikis you build from documents. Can be large. Read only when you ask for it, never loaded into a chat on its own.

These live under ~/.claude/mneme/ on your machine — the global memory that follows you everywhere. A project can also keep its own overlay in a .mneme folder inside that project, layered on top when you work there. Every note is a plain markdown file you can read with any text editor. Nothing leaves your computer: no cloud, no accounts, no tracking. To forget something, delete the file.

what you actually get

Both tiers are live today

Learning from your chats

Preferences, fixes, conventions, client facts, project context. You work, Mneme drafts, you approve, and the next chat is sharper. This is tier 1, the everyday memory.

Learning from your documents

Point it at a folder of papers, a long report, or a web page. It builds a wiki you can question across — with citations — that never crowds your everyday chats. This is tier 2, the knowledge base.

Honest about its edges

Your memory stays on the machine it was made on. There is no cross-machine sync and no shared team cache built in — though because every note is a plain file, you can commit a project's memory to git and share it that way. And the auto-drafted notes always wait for your approval; that gate is a feature, not a limitation.

the question a sharp user asks first

Why this, and not the memory that ships in the box

Claude Code already has CLAUDE.md (per-project, hand-maintained) and a built-in memory. So the real deltas have to be specific, or the pitch is hollow.

Plain markdownnot a black box
Notes are files you own: grep them, edit them, version them in git, move them between machines.
Discipline by designso it keeps loading
A relevance gate, dedupe, and pruning keep the index lean. Built-in memory bloats; Mneme has an opinion about staying small.
Global and projecttwo tiers of scope
A global cache that follows you everywhere, plus a per-repo overlay. Built-ins are essentially per-project.
Capture that quarantinesthe distiller
It drafts learnings into an inbox you approve from, never straight into your context.
An engine, not a fileskill bay + loop
A bay of skills that read and write the same cache, and /mneme:loop to drive a task to done.
The knowledge tierthe real reason
The wiki over your own documents is the one thing the built-ins flatly do not do. That is the use case that is only Mneme.
